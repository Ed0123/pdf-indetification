PDF 文本提取網絡應用 – 綜合文檔

概述 後端 API 功能流程 前端概覽 部署與需求

概述

此文檔在一頁中說明了整個 pdf‑identification 項目。它是通過閱讀 Python 和 TypeScript 源碼生成的，總結了每個公開入口的行為。

整個架構由託管在 Firebase Hosting 的 React + Vite 前端組成， 後端為 FastAPI 應用，部署於 Google Cloud Run。後端負責處理 PDF 上傳、渲染、文字提取、 項目狀態保存、模板存儲、導出（Excel/PDF/註釋 PDF）以及通過 Firebase 管理用戶帳戶。 系統還包含針對工程量清單的 BQ‑OCR 支持以及基於雲的模板管理功能。

後端 API 端點

POST /api/pdf/upload

觸發方式：multipart HTTP POST，上傳一個或多個 file 字段。

用途：接收 PDF 文件，暫存於磁盤並返回元數據。

輸入：文件列表。每個文件必須擴展名為 .pdf 且大小 ≤200 MB。

流程：

  1. upload_pdfs 驗證並寫入臨時目錄。
  2. 使用 PyMuPDF 打開每個文件以計算頁數；無效的 PDF 將被拒絕。
  3. 為每個文件生成 UUID 並存儲於內存字典 _STORE。
  4. 返回包含 FileInfo 對象的列表。

返回結果：JSON 數組，包含 file_id、 file_name、num_pages、file_size 以及 pages。 file_id 用於後續請求。

GET /api/pdf/{file_id}/page/{page_num}/render

觸發方式：HTTP GET，帶路徑參數 file_id 和 page_num，可選查詢 zoom。

用途：將指定頁面渲染為 base64 PNG 用於前端顯示。

輸入：有效的 file_id，0‑基礎 page_num。縮放因子在 0.1–5.0 之間，默認 1.5。

流程：render_page→_get_path→ utils.pdf_processing.render_pdf_page（調用 PyMuPDF 並轉為 PNG 字節）→ base64 編碼。

返回：JSON 包含 image（base64 字符串）和 page_num。

POST /api/pdf/{file_id}/page/{page_num}/extract

觸發方式：HTTP POST，JSON 主體含有邊界框座標和標誌。

用途：從矩形區域提取文本；優先矢量提取，必要時回退 OCR。

輸入：x,y,width,height 為相對座標 [0,1]，use_ocr 布爾值。

流程：extract_text→_get_path→ utils.pdf_processing.extract_text_from_relative_region→ 返回字符串。

返回：{"text": "..."} 包含提取的文本。

GET /api/pdf/{file_id}/page/{page_num}/dimensions

觸發方式：HTTP GET。

用途：獲取頁面尺寸（PDF 點）以便坐標換算。

輸入：file_id，page_num。

流程：page_dimensions→_get_path→ utils.pdf_processing.get_page_dimensions。

返回：{"width": float, "height": float}，若頁面不存在則 404。

POST /api/project/save

觸發方式：HTTP POST，原始 JSON 主體。

用途：將項目狀態流式返回給前端作為下載文件。

輸入：任意 JSON（項目狀態），大小 ≤50 MB。

流程：save_project讀取請求體，驗證大小和語法，返回帶有Content-Disposition的StreamingResponse。

返回：二進制 JSON 文件 project.json。

POST /api/project/load

觸發方式：multipart POST，包含單個 file 字段。

用途：接收之前保存的項目 JSON並返回其解析內容。

輸入：JSON 文件 ≤50 MB。

流程：load_project讀取文件，檢查擴展名和大小，解析 JSON並返回對象。

返回：反序列化的 JSON 或錯誤代碼 422/413。

POST /api/export/excel

觸發方式：HTTP POST，JSON 載荷。

用途：從完整項目狀態生成 Excel 工作簿。

輸入：與 save/load 相同的項目 JSON。

流程：export_excel→通過 models.data_models 重建 ProjectData→utils.excel_export.export_to_excel 寫入臨時文件→流式返回並刪除。

返回：application/vnd.openxmlformats-officedocument.spreadsheetml.sheet，文件名 export.xlsx。

PDF 匯出端點

這兩個端點位於 /api/pdf/export-pages 和 /api/pdf/export-annotated。 支持將所選頁面匯出為單獨 PDF 或帶註釋的文檔。完整參數模型見源碼。

用戶和管理員端點

一組路由在 /api/users 下處理個人資料獲取（GET /me）、更新（PUT /me）、管理員列表與編輯， 以及管理群組、等級和記錄使用情況的輔助方法。所有調用均需要 Firebase 令牌，並調用輔助函數確保默認資料和權限。

模板管理端點

在 /api/templates 下實現 CRUD 操作，用於在 Firestore 中存儲提取模板。 輔助函數計算用戶群、管理員狀態和雲存儲路徑；頁面圖像可以通過 multipart 或 base64 方式上傳， 檢索時返回簽署 URL 或內聯 base64 數據。權限（個人/公開/群組）受到強制執行，管理員可覆蓋。

GET /api/bq/engines

觸發方式：HTTP GET。

用途：列出可用的工程量清單 OCR 引擎（pdfplumber、camelot 等），並指示當前環境中哪些可用。

返回：包含 id,name,quota_cost,available,description 的對象數組。

POST /api/bq/extract

觸發方式：HTTP POST，主體為 JSON。

用途：使用提供的框定義和選定引擎對一頁或多頁執行 BQ‑OCR 提取。

輸入：file_id、pages 列表、含 column_name,x,y,width,height 的 boxes 區，及可選 engine。

返回：success 標誌和解析行的 rows 陣列，以及使用成本和任何警告。

BQ 模板端點

在 /api/bq/templates 下的 CRUD 操作，用於存儲提取版式。支持通過 base64 或 multipart 上傳頁面圖像，並返回預覽的簽名 URL。 權限根據個人/公開/群組作用域及管理員覆蓋。

所有 API 端點均由 backend/auth_middleware.py 中的 require_auth 保護， 驗證 Firebase 令牌。backend/main.py 的 startup 事件初始化 Firebase，並在 /api/health 和 /api/ocr/status 暴露健康與 OCR 狀態端點。

功能流程（詳細步驟）

本節描述最常用工作流的內部呼叫順序。

PDF 上傳與渲染/提取流程

  1. 前端 uploadPdfs() 將文件發送至 /api/pdf/upload。
  2. 後端 upload_pdfs 存儲文件並更新 _STORE。
  3. 渲染頁面時前端使用 render_page 端點，該端點調用 utils.pdf_processing.render_pdf_page 回傳 PNG 字節。
  4. 執行提取請求時觸發 extract_text，其委派給 utils.pdf_processing.extract_text_from_relative_region。該輔助函數 在 PyMuPDF 打開 PDF 並嘗試文本提取；如果結果為空且 use_ocr 為真，則回退調用 utils.pdf_processing.do_ocr。
  5. 結果返回前端並填充數據表。

項目保存/加載與導出流程

  1. 用戶在 UI 觸發"保存項目"。前端將完整狀態 POST 到 /api/project/save，在那裡 save_project 驗證 JSON 並以 適當頭部流式返回。
  2. 使用"加載項目"時，前端將文件發送到 /api/project/load；後端反序列化並返回對象。
  3. Excel 導出將相同項目 JSON POST 到 /api/export/excel；後端 從 models/data_models.py 重建 ProjectData，然後調用 utils.excel_export.export_to_excel，該函數遍歷頁面/模板/數據並使用 openpyxl 寫入單元格。生成的工作簿被流式返回。

模板 CRUD 流程

  1. 前端組件如 TemplateManagerPanel 調用相應 /api/templates 端點。
  2. list_templates 查詢 Firestore 並按權限、群組和管理員狀態過濾。
  3. 創建和更新直接操作 Firestore 文檔，添加時間戳、所有者 UID，並將模型對象轉換為字典。
  4. 頁面圖像上傳使用雲存儲；multipart 或 base64 均可進行。 檢索時嘗試從存儲下載，並在需要時返回內聯 base64。

用戶與管理員管理

身份驗證由 Firebase 處理；請求到達時令牌被驗證， require_auth 返回包含 uid、電子郵件、名稱等的字典。 用戶路由確保配置個人資料並緩存等級/群組資訊。管理員路由使用 _is_admin 進行守衛並可能通過 backend/email_service.py 觸發電子郵件通知。

前端概覽

客戶端使用 TypeScript + React 編寫；關鍵組件包括：

  • App.tsx – 根組件，處理路由和認證狀態。
  • PDFTreeView.tsx、PDFViewer.tsx – 顯示頁面並接受提取框繪製。
  • DataTable.tsx/ SinglePageDataTable.tsx – 顯示提取文本並允許編輯。
  • Toolbar.tsx、StatusBar.tsx – 用戶操作和進度。
  • API 客戶端 api/client.ts 封裝 `fetch` 並附加 Firebase 令牌。鉤子 `useProject.ts` 協調當前 PDF/框/模板/數據的狀態管理。

每個 UI 操作的啟動方法與上述後端路由緊密對應。例如，單擊 “識別文字”會觸發對 extract 的調用，成功後最終更新狀態。 前端代碼是自包含的；發佈時，轉譯後的資源位於 frontend-dist/。

所有前端依賴列於 frontend/package.json；開發期間運行 npm run dev 可在 http://localhost:5173 啟動 Vite 服務並透過代理 連接後端。

部署與需求

項目提供一條命令腳本（deploy.sh）構建後端和前端，並分別部署到 Cloud Run 和 Firebase Hosting。手動步驟記錄在原始的 README.md (點此)。
